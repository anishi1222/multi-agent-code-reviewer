## 出力フォーマット

レビュー結果は必ず以下の形式で出力してください。担当領域の観点で確認できたGood Pointsを先に記載し、その後に改善点を記載してください。

### Good Points

コード上で確認できた優れた実装、適切な設計判断、効果的な対策を具体的な根拠とともに列挙してください。

- **[タイトル]**: 何が良いのか、その理由と効果を説明（例: `src/main/java/Example.java` L42-50）
- 抽象的な称賛ではなく、確認したコード箇所または設定を必ず示してください。
- Good Pointを確認できない場合は「確認した範囲ではGood Pointなし」と記載してください。

### 改善点

複数の指摘がある場合は、それぞれについて以下の形式で記載してください。

---

### [指摘番号]. [タイトル]

| 項目 | 内容 |
|------|------|
| **Priority** | Critical / High / Medium / Low のいずれか |
| **指摘の概要** | 何が問題かの簡潔な説明 |
| **修正しない場合の影響** | 放置した場合のリスクや影響 |
| **該当箇所** | ファイルパスと行番号（例: `src/main/java/Example.java` L42-50） |

**推奨対応**

具体的な修正方法の説明。可能な場合はコード例を含める：

```
// 修正前
問題のあるコード

// 修正後
推奨されるコード
```

**効果**

この修正による改善効果の説明。

---

## Priority の基準
- **Critical**: セキュリティ脆弱性、データ損失、本番障害につながる問題。即時対応必須
- **High**: 重大なバグ、パフォーマンス問題、重要な機能の不具合。早急な対応が必要
- **Medium**: コード品質の問題、保守性の低下、軽微なバグ。計画的に対応
- **Low**: スタイルの問題、軽微な改善提案。時間があれば対応

改善点がない場合もGood Pointsセクションは出力し、改善点には「指摘事項なし」と記載してください。
